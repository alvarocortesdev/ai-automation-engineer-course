En [4.5 React + TypeScript](/fase-4-frontend/4-5-react-typescript/) aprendiste a manejar el estado que vive **dentro** de tu app: un contador, el texto de un input, una pestaña activa. Pero la mayoría de las apps reales muestran datos que **no son tuyos**: viven en una base de datos, detrás de la API que construiste en la Fase 3, y llegan por la red. Esos datos son otra bestia. Llegan tarde, fallan, se quedan viejos, los comparten varias pantallas, y mientras esperas tienes que dibujar "cargando". Esta lección trata sobre la distinción más importante del frontend moderno —**server state vs client state**— y sobre las dos herramientas que en 2026 todo equipo serio usa para domarlos: **TanStack Query** para los datos del servidor y **React Hook Form + zod** para los formularios. La enseña desde cero, sin asumir que ya tocaste ninguna de las dos.

> La trampa de esta lección: pensar que traer datos del servidor es "solo un `fetch` dentro de un `useEffect`". Lo es, en el sentido en que cruzar el océano es "solo nadar". Funciona en la demo del tutorial y se desmorona en cuanto hay dos pantallas que muestran lo mismo, una conexión lenta, o un usuario que hace clic dos veces. El 80% del código frágil de un frontend junior es **server state manejado a mano**. Aprender a delegarlo es lo que separa una UI que se siente sólida de una que parpadea, se desincroniza y pierde datos.

## 2. Por qué importa (el dinero está aquí)

> 💰 **Por qué importa:** un AI Engineer que monta la UI de su propia demo vale más que el que depende de alguien. Pero la demo se cae en el primer detalle de datos reales: el chat que muestra mensajes viejos, el botón "guardar" que dispara dos requests, la lista que no se actualiza tras crear un item. Saber **manejar server state** es exactamente lo que convierte una demo de juguete en algo que un cliente usaría. Y es pregunta de entrevista garantizada: "¿por qué no usas `useEffect` para traer datos?".

Concreto, y sin vender humo:

- **El server state es el problema central del frontend, y casi nadie lo nombra.** Caché, deduplicación de requests, refetch al volver a la pestaña, reintentos, datos viejos vs frescos: TanStack Query resuelve todo eso con una API pequeña. Reescribirlo a mano es la fuente #1 de bugs y de código que no se puede mantener.
- **Optimistic updates es lo que hace que una app "se sienta rápida".** Cuando marcas una tarea como hecha y la UI responde al instante (sin esperar al servidor) pero **sabe deshacer el cambio si el servidor falla**, eso es ingeniería de frontend de nivel. Es el patrón detrás de cada app que se siente fluida.
- **Los formularios son donde entran los datos sucios.** Validar en el cliente con el **mismo contrato** que el servidor (zod en el cliente espejando tu modelo de pydantic en FastAPI) es defensa en profundidad: el usuario recibe feedback inmediato y el servidor nunca confía en el cliente. Esto conecta directo con OWASP (validación de entrada) que ya viste en el backend.
- **Es la mitad del capstone.** El [Capstone F4](/fase-4-frontend/proyecto/) es una UI de chat que consume tu backend de la Fase 3: la lista de mensajes es una query, enviar un mensaje es una mutación (optimista), y el formulario de entrada se valida con zod. Esta lección **es** ese capstone.

## 4. Ejemplo resuelto, pensado en voz alta

Voy a construir la capa de datos de una mini-app de tareas que consume el backend de la Fase 3, razonando en voz alta. **No lo leas como reglas sueltas: léelo como me oirías pensar al lado tuyo.** Empiezo por el malentendido que hay que matar, y de ahí sale todo lo demás.

### 4.1 La distinción que lo cambia todo: server state vs client state

Pienso en voz alta: *"Antes de tocar una librería, separo el estado de mi app en dos cubetas, porque se comportan de forma completamente distinta."*

| | **Client state** | **Server state** |
|---|---|---|
| Quién es dueño | Tu componente | El servidor (la DB) |
| Cuándo está disponible | Al instante (síncrono) | Tarde, por la red (asíncrono) |
| Puede fallar | No | **Sí** (red, 500, timeout) |
| Se queda viejo | No | **Sí** (otro usuario lo cambió) |
| Lo comparten varias pantallas | Raro | **Casi siempre** |
| Herramienta | `useState` / `useReducer` / Zustand | **TanStack Query** |

*"El texto de un input, una pestaña activa, si un modal está abierto: eso es client state, vive y muere en mi UI, `useState` basta. Pero la lista de tareas que viene de `GET /tareas` es server state: no es mía, llega tarde, puede fallar, y si la muestro en dos lugares quiero que ambos vean lo mismo. Tratar server state como si fuera client state —guardarlo en un `useState` y sincronizarlo a mano— es el error de origen. Necesita una herramienta que entienda que es una **copia en caché de algo remoto**."*

### 4.2 El antipatrón: `fetch` en `useEffect` (y por qué se rompe)

*"Casi todos empiezan así. Voy a escribirlo y luego a contar todo lo que le falta."*

```tsx
// 🐛 El patrón que parece razonable y se desmorona en producción.
function ListaTareas() {
  const [tareas, setTareas] = useState<Tarea[]>([]);
  const [cargando, setCargando] = useState(true);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    setCargando(true);
    fetch("/api/tareas")
      .then((r) => r.json())
      .then((data) => setTareas(data))
      .catch((e) => setError(String(e)))
      .finally(() => setCargando(false));
  }, []);

  if (cargando) return <p>Cargando...</p>;
  if (error) return <p>Error: {error}</p>;
  return <ul>{tareas.map((t) => <li key={t.id}>{t.titulo}</li>)}</ul>;
}
```

*"Funciona en la demo. Pero cuento lo que NO hace, y cada punto es código que tendría que escribir a mano:"*

1. **No hay caché ni deduplicación.** Si dos componentes piden `/api/tareas`, hago dos requests. Si navego fuera y vuelvo, pido todo de nuevo desde cero, pantalla en blanco incluida.
2. **No hay manejo de race conditions.** Como vimos en 4.5, si la dependencia cambia rápido, las respuestas pueden llegar desordenadas. Aquí falta el `cleanup` con flag, y casi nadie lo pone.
3. **No hay reintentos ni refetch.** Si el request falla por un parpadeo de red, se queda en error para siempre. No vuelve a intentar, no refresca al volver a la pestaña.
4. **No hay forma de invalidar.** Cuando creo una tarea nueva con `POST`, ¿cómo le digo a esta lista "estás vieja, vuelve a pedir"? A mano: levantar el estado, pasar callbacks, un enredo.
5. **El boilerplate se repite en cada pantalla.** Tres `useState` (`datos`/`cargando`/`error`) y un efecto, copiados una y otra vez. Es ruido que oculta la lógica real.

*"TanStack Query me regala las cinco cosas gratis. Por eso no es 'una librería más': es la herramienta correcta para una categoría de estado que React por sí solo no maneja."*

### 4.3 `useQuery`: leer datos del servidor

*"Primero, una vez en la raíz de la app, instalo el proveedor de caché. Es como el `<Provider>` de cualquier contexto: crea la caché que todas las queries comparten."*

```tsx
// main.tsx — se hace UNA vez, en la raíz.
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";

const queryClient = new QueryClient();

createRoot(document.getElementById("root")!).render(
  <QueryClientProvider client={queryClient}>
    <App />
  </QueryClientProvider>,
);
```

*"Ahora la query. Necesito una `queryFn` que traiga los datos y —detalle crítico— que **lance** (throw) si la respuesta no es OK, porque TanStack Query detecta el error por la excepción."*

```tsx
interface Tarea {
  id: string;
  titulo: string;
  hecha: boolean;
}

// La función de fetch: lanza si el servidor responde mal.
async function obtenerTareas(): Promise<Tarea[]> {
  const res = await fetch("/api/tareas");
  if (!res.ok) throw new Error(`HTTP ${res.status}`); // ← clave: sin esto isError nunca se activa
  return res.json();
}

function ListaTareas() {
  const { data, isPending, isError, error } = useQuery({
    queryKey: ["tareas"],   // identidad de esta query en la caché
    queryFn: obtenerTareas, // cómo se traen los datos
  });

  if (isPending) return <p>Cargando tareas...</p>;
  if (isError) return <p role="alert">Error: {error.message}</p>;

  return (
    <ul>
      {data.map((t) => <li key={t.id}>{t.titulo}</li>)}
    </ul>
  );
}
```

Tres cosas que decir despacio:

- **`queryKey` es la identidad del dato en la caché.** `["tareas"]` es una lista; `["tareas", id]` sería una tarea concreta. Dos componentes con la misma key comparten **una** petición y **un** resultado: ahí está la deduplicación y la caché, gratis.
- **`isPending` (no `isLoading`).** En TanStack Query v5 el estado inicial sin datos se llama `isPending`. Después de ese guard, TypeScript **sabe** que `data` ya no es `undefined`: por eso `data.map(...)` compila sin un `?`. Los guards ordenan los tipos.
- **`queryFn` debe lanzar en error.** Si te tragas el error y devuelves `[]`, `isError` nunca se activa y la UI miente diciendo "no hay tareas" cuando en realidad el servidor está caído.

### 4.4 Caché, datos viejos (`staleTime`) e invalidación

*"La idea mental de la caché: un dato recién traído está **fresco** (`fresh`); pasado un tiempo (`staleTime`) pasa a **viejo** (`stale`). Viejo no significa borrado: se sigue mostrando al instante desde la caché, pero TanStack Query lo **refresca en segundo plano** en ciertos disparadores (volver a la pestaña, reconectar la red, montar el componente). Por defecto `staleTime` es 0: todo se considera viejo de inmediato y se refresca agresivamente. Para datos que no cambian cada segundo, subir `staleTime` evita refetches innecesarios."*

```tsx
useQuery({
  queryKey: ["tareas"],
  queryFn: obtenerTareas,
  staleTime: 30_000, // 30 s frescos: no refetchea de más en ese rato
});
```

*"¿Y cuándo YO sé que el dato cambió —porque acabo de crear una tarea? No espero a que `staleTime` venza: **invalido** la query a mano. Invalidar marca esa key como vieja y dispara un refetch. Esta es la pieza que el `useEffect`-fetch no tenía."*

```tsx
import { useQueryClient } from "@tanstack/react-query";

const queryClient = useQueryClient();
// "la lista de tareas quedó desactualizada, vuelve a pedirla":
queryClient.invalidateQueries({ queryKey: ["tareas"] });
```

### 4.5 `useMutation`: escribir, y luego invalidar

*"Leer es `useQuery`; crear/editar/borrar es `useMutation`. La mutación no se cachea (no es un dato, es una acción), pero me da `isPending`/`isError` y, sobre todo, ganchos de ciclo de vida. El patrón base: muto en el servidor, y al terminar **invalido** la lista para que se refresque sola."*

```tsx
async function crearTarea(titulo: string): Promise<Tarea> {
  const res = await fetch("/api/tareas", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ titulo }),
  });
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

function FormularioRapido() {
  const queryClient = useQueryClient();
  const mutacion = useMutation({
    mutationFn: crearTarea,
    onSuccess: () => {
      // El servidor confirmó: la lista quedó vieja → refréscala.
      queryClient.invalidateQueries({ queryKey: ["tareas"] });
    },
  });

  return (
    <button
      disabled={mutacion.isPending}
      onClick={() => mutacion.mutate("Nueva tarea")}
    >
      {mutacion.isPending ? "Creando..." : "Crear"}
    </button>
  );
}
```

*"`invalidateQueries` (refetch tras confirmar) es la opción por defecto, segura: la UI siempre termina mostrando lo que el servidor realmente tiene. Pero tiene un costo: hay un instante entre el clic y la actualización. Para acciones frecuentes (marcar hecho, dar like) ese retraso se nota. Ahí entra el optimistic update."*

### 4.6 Optimistic updates con rollback (el patrón que se siente rápido)

*"La idea: actualizo la caché **antes** de que el servidor responda, asumiendo que va a salir bien. Si sale bien, perfecto, ya estaba pintado. Si falla, **deshago** (rollback) al valor anterior. El truco está en los tres ganchos: `onMutate` (antes: actualizo y guardo el valor previo), `onError` (si falla: restauro), `onSettled` (pase lo que pase: invalido para acabar sincronizado con el servidor)."*

```tsx
function useToggleTarea() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: toggleTareaApi, // PATCH /api/tareas/{id} → devuelve la tarea
    // 1) ANTES de llamar al servidor:
    onMutate: async (id: string) => {
      // Cancelo refetches en vuelo para que no pisen mi update optimista.
      await queryClient.cancelQueries({ queryKey: ["tareas"] });
      // Snapshot del valor actual, por si hay que deshacer.
      const previas = queryClient.getQueryData<Tarea[]>(["tareas"]);
      // Actualizo la caché optimistamente (inmutable: copio, no muto).
      queryClient.setQueryData<Tarea[]>(["tareas"], (viejas) =>
        (viejas ?? []).map((t) =>
          t.id === id ? { ...t, hecha: !t.hecha } : t,
        ),
      );
      // Lo que retorno aquí llega como `contexto` a onError/onSettled.
      return { previas };
    },
    // 2) Si el servidor falla: restauro el snapshot.
    onError: (_err, _id, contexto) => {
      if (contexto?.previas) {
        queryClient.setQueryData(["tareas"], contexto.previas);
      }
    },
    // 3) Pase lo que pase: invalido para terminar igual al servidor.
    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ["tareas"] });
    },
  });
}
```

*"Léelo como un seguro: `onMutate` apuesta a que va a funcionar y guarda el recibo; `onError` cobra el seguro y revierte; `onSettled` reconcilia con la verdad del servidor. El detalle que casi todos olvidan es `cancelQueries`: sin él, un refetch que ya estaba en vuelo puede aterrizar con datos viejos justo después de tu update y deshacerlo visualmente. Y la actualización es **inmutable** (`.map` que crea un array nuevo, spread `{ ...t }`): mutar la caché en sitio rompe la detección de cambios de React, igual que con `useState`."*

### 4.7 Formularios = client state: React Hook Form + zod

*"Cambio de cubeta. El texto que el usuario escribe en un formulario **mientras** lo llena es client state puro: vive en la UI, no ha tocado el servidor. ¿Por qué no un `useState` por campo? Porque un formulario de verdad tiene validación por campo, mensajes de error, estado 'enviando', y cada tecleo en un input controlado re-renderiza todo. React Hook Form usa inputs **no controlados** (lee con refs) para no re-renderizar en cada tecla, y delega la validación a un esquema. Con zod, ese esquema es además mi tipo de TypeScript."*

```tsx
import { useForm } from "react-hook-form";
import { zodResolver } from "@hookform/resolvers/zod";
import { z } from "zod";

// El esquema ES la fuente de verdad: valida Y define el tipo.
const esquemaTarea = z.object({
  titulo: z.string().min(1, "El título es obligatorio").max(80, "Máximo 80 caracteres"),
  prioridad: z.enum(["baja", "media", "alta"]),
  estimacionMin: z.number().int().positive("Debe ser un número positivo"),
});

type NuevaTarea = z.infer<typeof esquemaTarea>; // { titulo: string; prioridad: ...; estimacionMin: number }

function FormularioTarea({ onCrear }: { onCrear: (t: NuevaTarea) => void }) {
  const {
    register,
    handleSubmit,
    formState: { errors, isSubmitting },
  } = useForm<NuevaTarea>({
    resolver: zodResolver(esquemaTarea),
    defaultValues: { prioridad: "media" },
  });

  return (
    <form onSubmit={handleSubmit(onCrear)} noValidate>
      <label htmlFor="titulo">Título</label>
      <input id="titulo" {...register("titulo")} />
      {errors.titulo && <p role="alert">{errors.titulo.message}</p>}

      <label htmlFor="estimacionMin">Estimación (min)</label>
      {/* valueAsNumber: el input devuelve string; esto lo convierte a number para zod */}
      <input id="estimacionMin" type="number" {...register("estimacionMin", { valueAsNumber: true })} />
      {errors.estimacionMin && <p role="alert">{errors.estimacionMin.message}</p>}

      <button type="submit" disabled={isSubmitting}>Crear tarea</button>
    </form>
  );
}
```

Cosas que decir despacio:

- **`handleSubmit(onCrear)`** corre el resolver primero. Si la validación falla, **`onCrear` no se llama**: los errores aparecen en `errors` y listo. Tu callback solo recibe datos **ya válidos y tipados** como `NuevaTarea`.
- **`{ valueAsNumber: true }`** es el detalle que muerde a todos: un `<input type="number">` devuelve un **string** (`"30"`), y `z.number()` lo rechazaría. Esa opción lo convierte a `number` antes de validar.
- **`z.infer`** evita declarar el tipo dos veces. El esquema valida en runtime y, además, **es** el tipo en tiempo de compilación. Una sola fuente de verdad.
- **`z.email()`** es la forma de validar email en zod v4 (la vieja `z.string().email()` quedó deprecada). No la usamos aquí, pero tenla presente.

### 4.8 Reusar el contrato zod con el backend

*"Pregunta honesta: ¿puedo usar literalmente el mismo zod en cliente y servidor? Depende del stack."*

- **Si el backend es TypeScript** (Next.js Route Handlers, como verás en [4.6 Next.js](/fase-4-frontend/4-6-nextjs/)): sí, pones `esquemaTarea` en un archivo compartido y lo importas en ambos lados. **Una** definición valida el formulario en el cliente y el body en el servidor. Eso es el sueño de un solo contrato.
- **Si el backend es FastAPI** (tu Fase 3, Python): no puedes importar pydantic en TS. Pero el contrato es el mismo modelo en dos idiomas. La forma profesional de mantenerlos en sync no es copiar a mano: es **generar** los tipos del cliente desde el **OpenAPI schema** que FastAPI publica (herramientas como `openapi-zod-client` u `orval`). Así, si el backend cambia un campo, el cliente deja de compilar. El espíritu es el mismo de spec-driven: una sola fuente de verdad para el contrato.

*"En cualquier caso, la regla de seguridad no cambia: validar en el cliente es para UX (feedback inmediato); el servidor **siempre** re-valida. Nunca confíes en que el cliente mandó datos limpios —es exactamente OWASP de la Fase 3 aplicado al frontend."*

## 5. Errores y malentendidos comunes

:::caution[Podrías pensar... pero está mal]

**"TanStack Query es un cliente HTTP, reemplaza a `fetch`/`axios`."**
Mal. No hace requests por ti: tú escribes la `queryFn` con `fetch`. Lo que gestiona es el **estado asíncrono** alrededor de esos datos: caché, staleness, reintentos, invalidación, deduplicación. Es un *manejador de server state*, no un cliente HTTP.

**"Tras una mutación, actualizo la lista llamando `setQueryData` a mano para no refetchear."**
Casi siempre prefiere `invalidateQueries`. `setQueryData` manual asume que tu cálculo del nuevo estado coincide exactamente con lo que el servidor guardó (¿y si el servidor calculó un campo, asignó un id, aplicó una regla?). Invalidar y refetchear te da la **verdad del servidor**. El `setQueryData` manual solo se justifica en el optimistic update (donde es temporal y `onSettled` invalida igual).

**"Guardo la respuesta de `useQuery` en un `useState` para 'tenerla a mano'."**
El antipatrón clásico. Copiar server state a un `useState` lo congela: cuando la query refresca, tu copia se queda vieja. Los datos del servidor viven en la caché de TanStack Query; léelos desde `data`, no los dupliques.

**"Hago el input controlado con `useState` y valido a mano en el `onSubmit`."**
Funciona para un campo, pero no escala: re-renderizas en cada tecla, reinventas mensajes de error, estados touched/dirty, y la validación se desincroniza del tipo. RHF + zod te da todo eso y el esquema **es** el tipo.

**"`valueAsNumber` es opcional, igual valida."**
No. Sin él, un `<input type="number">` entrega el string `"30"` y `z.number()` falla con "expected number, received string". Es el bug #1 de formularios numéricos con RHF + zod.

**"Validar en el cliente con zod ya protege el backend."**
No. El cliente es manipulable (DevTools, curl directo a tu API). La validación de cliente es UX; el servidor **debe** re-validar siempre. Defensa en dos capas.

:::

Un *non-example* que parece correcto pero no lo es —léelo y detecta el bug antes de seguir:

```tsx
// 🐛 ¿Qué tiene de malo este "optimistic update"?
function useBorrarTarea() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: borrarTareaApi,
    onMutate: (id) => {
      queryClient.setQueryData<Tarea[]>(["tareas"], (viejas) =>
        (viejas ?? []).filter((t) => t.id !== id),
      );
      // (sin cancelQueries, sin snapshot, sin return)
    },
    onError: () => {
      // ¿restaurar a qué? no guardé nada
    },
  });
}
```

Tiene tres bugs: (1) **no hay snapshot** del valor previo, así que `onError` no tiene a qué volver —si el `DELETE` falla, la tarea desaparece para siempre de la UI aunque siga en la DB—. (2) **falta `cancelQueries`**: un refetch en vuelo puede repintar la tarea borrada justo después. (3) **falta `onSettled` con invalidación**: la UI nunca se reconcilia con el servidor. El patrón completo de la sección 4.6 los cubre los tres; este es **exactamente** el músculo que entrenas en el ejercicio A.

## 6. Práctica con andamiaje (PRIMM)

Antes de construir desde cero, practica con código que ya existe, usando el ciclo **PRIMM**: *Predict → Run → Investigate → Modify → Make*.

### 6.1 Predice (antes de ejecutar nada)

Lee esta mutación y **predice en voz alta o en papel** qué se ve en pantalla en los dos escenarios:

```tsx
const mutacion = useMutation({
  mutationFn: crearTarea, // POST que tarda ~1s
  onSuccess: () => queryClient.invalidateQueries({ queryKey: ["tareas"] }),
});
// El botón: <button onClick={() => mutacion.mutate("X")}>Crear</button>
// La lista de arriba usa useQuery(["tareas"]).
```

Preguntas: (1) Justo tras el clic y **antes** de que el `POST` responda, ¿aparece ya "X" en la lista? (2) ¿Cuándo aparece "X"?

<Solucion title="Ver respuesta (después de predecir)">

(1) **No** aparece de inmediato: esto NO es optimista. La caché de `["tareas"]` no se toca hasta que la mutación termina. (2) "X" aparece **después** de que el `POST` responde con éxito → `onSuccess` invalida → TanStack Query refetchea `GET /tareas` → llega la lista nueva (que ya incluye "X"). Hay un retraso visible de ~1s + el tiempo del refetch. Para que aparezca al instante, necesitarías un optimistic update (sección 4.6). Este es el trade-off `invalidateQueries` (simple y siempre correcto) vs optimistic (rápido pero más código).

</Solucion>

### 6.2 Parsons: reordena los ganchos de un optimistic update

Estas líneas forman el `onMutate` de un optimistic update, pero están **desordenadas**. Reescríbelas en el orden correcto (a mano):

```text
A)   const previas = queryClient.getQueryData<Tarea[]>(["tareas"]);
B)   return { previas };
C)   await queryClient.cancelQueries({ queryKey: ["tareas"] });
D)   queryClient.setQueryData<Tarea[]>(["tareas"], (v) => /* ...update... */);
```

<Solucion title="Ver orden correcto (después de intentarlo)">

`C → A → D → B`:

1. **`C` cancelar** los refetches en vuelo (para que no pisen el update).
2. **`A` snapshot** del valor actual (para poder revertir).
3. **`D` actualizar** la caché optimistamente.
4. **`B` retornar** el snapshot como contexto para `onError`/`onSettled`.

El orden importa: si actualizaras (`D`) antes de tomar el snapshot (`A`), guardarías el valor **ya modificado** y el rollback no serviría de nada. Y cancelar (`C`) debe ir primero, antes de tocar la caché.

</Solucion>

## 10. Conexión con el capstone

El [Capstone F4 — Frontend de una app de IA](/fase-4-frontend/proyecto/) **es** esta lección aplicada sobre tu backend de la Fase 3:
