Hasta aquí pintabas pantallas con HTML, CSS y Tailwind: marcado estático que tú escribes una vez. React es otra cosa. Es una forma de describir **qué se ve en pantalla en función del estado**, y dejar que la librería se encargue de actualizar el DOM cuando ese estado cambia. Tú no dices "cambia este `<span>`": dices "cuando el contador vale 5, la pantalla se ve así", y React calcula la diferencia. Ese cambio de mentalidad —de *manipular el DOM* a *declarar la UI*— es todo el juego. Esta lección lo enseña desde cero, con TypeScript pegado desde la primera línea, porque en 2026 React sin tipos es código que nadie en un equipo serio va a aprobar.

> La trampa de esta lección: tratar a React como "HTML con superpoderes" y a los hooks como cajas mágicas que copias de Stack Overflow. El 80% de los bugs de React de un junior salen de **no entender cuándo se vuelve a renderizar un componente** y de **meter en `useEffect` cosas que no son efectos**. Si entiendes esos dos modelos mentales, el resto son detalles de sintaxis.

## 1. Qué vas a saber hacer

Al terminar, sin IA y sin notas, podrás:

- **O1 — Implementar** un componente funcional en React 19 + TypeScript con **props tipadas**, estado con `useState` y un **input controlado**, renderizando una lista con `key` estables.
- **O2 — Explicar y aplicar** el modelo de re-render de React: qué dispara un render, por qué el **estado derivado se calcula durante el render** (no en un `useEffect`), y cuándo `useMemo`/`useCallback` importan de verdad.
- **O3 — Depurar** los antipatrones de `useEffect`: dependencias faltantes o sobrantes, falta de *cleanup* que provoca *race conditions*, y efectos que en realidad deberían ser estado derivado o *event handlers*.

## 4. Ejemplo resuelto, pensado en voz alta

Te voy a construir un componente real de principio a fin, razonando en voz alta. **No lo leas como reglas sueltas: léelo como me oirías pensar al lado tuyo.** Vamos a hacer un panel mínimo para una app de IA: un selector de modelo y un contador de mensajes enviados. Empiezo por lo más simple y voy agregando capas, explicando cada decisión.

### 4.1 El componente más pequeño posible

Pienso en voz alta: *"Un componente de React es solo una función que devuelve JSX. JSX parece HTML pero es JavaScript: el compilador lo convierte en llamadas a funciones. La función recibe sus datos de entrada como un único argumento, `props`, y yo lo destructuro."*

```tsx
// Un componente es una función que devuelve JSX. Nombre en PascalCase (obligatorio:
// si empieza en minúscula, React lo trata como una etiqueta HTML literal).
function Saludo() {
  return <h2>Panel de IA</h2>;
}
```

JSX no es HTML: fíjate que más abajo usaré `className` en vez de `class`, y `onClick` en vez de `onclick`. Son props de JavaScript, en camelCase. Y todo lo que va entre `{ }` dentro del JSX es **una expresión de JavaScript** que se evalúa: `{2 + 2}` renderiza `4`.

### 4.2 Props tipadas: el contrato de entrada

*"Quiero que el panel reciba de afuera la lista de modelos disponibles y un título. Eso son sus props. En TypeScript defino una `interface` que describe la forma exacta de esas props: es un contrato. Si alguien usa `<Panel />` sin pasar `modelos`, el compilador se queja antes de ejecutar nada."*

```tsx
interface PanelProps {
  titulo: string;
  modelos: string[];        // arreglo de nombres de modelo
  porDefecto?: string;      // el `?` lo hace opcional
}

function Panel({ titulo, modelos, porDefecto }: PanelProps) {
  return (
    <section>
      <h2>{titulo}</h2>
      <p>{modelos.length} modelos disponibles</p>
    </section>
  );
}
```

Esto es lo mismo que escribir una mini-spec de la función antes de codearla, solo que el compilador la verifica por ti. La interface `PanelProps` es la documentación viva del componente.

### 4.3 Estado con `useState`: lo que cambia en el tiempo

*"Las props vienen de afuera y el componente no las modifica. Pero el modelo seleccionado y el número de mensajes cambian con la interacción del usuario: eso es **estado**. Para que React vuelva a dibujar la pantalla cuando el estado cambia, no puedo usar una variable normal —una variable normal se reinicia en cada render y React no se entera de que cambió—. Uso el hook `useState`."*

```tsx
import { useState } from "react";

function Panel({ titulo, modelos, porDefecto }: PanelProps) {
  // useState devuelve [valorActual, funciónParaActualizarlo].
  // El tipo del estado se infiere del valor inicial; aquí lo anoto explícito.
  const [modelo, setModelo] = useState<string>(porDefecto ?? modelos[0]);
  const [enviados, setEnviados] = useState(0);

  return (
    <section>
      <h2>{titulo}</h2>
      <p>Modelo activo: {modelo}</p>
      <p>Mensajes enviados: {enviados}</p>
      <button onClick={() => setEnviados(enviados + 1)}>Enviar</button>
    </section>
  );
}
```

Modelo mental clave, y di esto despacio: **cada llamada a una función `setX` le pide a React que re-ejecute la función del componente con el nuevo valor.** No es que React "vigile" la variable; es que `setEnviados` agenda un nuevo render, y en ese render `useState` te devuelve el valor actualizado. El estado es una **foto fija** dentro de cada render: durante un render dado, `enviados` no cambia de valor aunque llames a `setEnviados` mil veces.

### 4.4 La trampa del estado como "foto": actualizadores funcionales

*"Si quiero subir el contador de dos en dos con un solo clic, lo intuitivo es llamar dos veces a `setEnviados(enviados + 1)`. Pero falla: ambas llamadas leen el mismo `enviados` (la foto de este render, digamos 0), así que las dos calculan 1. El contador sube a 1, no a 2."*

La solución es pasarle a `setX` una **función actualizadora** que recibe el valor más reciente:

```tsx
// ❌ Sube solo 1: ambas leen el mismo `enviados` de este render.
setEnviados(enviados + 1);
setEnviados(enviados + 1);

// ✅ Sube 2: cada actualizador recibe el resultado del anterior.
setEnviados((n) => n + 1);
setEnviados((n) => n + 1);
```

Esta es de las preguntas de entrevista más comunes sobre React. La regla práctica: **si el nuevo estado depende del anterior, usa la forma funcional `setX(prev => ...)`.**

### 4.5 Listas y `key`: renderizar un arreglo

*"Quiero mostrar todos los modelos como botones para elegir. Mapeo el arreglo de strings a un arreglo de elementos JSX. React me va a pedir un `key` por elemento."*

```tsx
<ul>
  {modelos.map((nombre) => (
    <li key={nombre}>
      <button onClick={() => setModelo(nombre)}>{nombre}</button>
    </li>
  ))}
</ul>
```

El `key` no es decorativo: es **cómo React identifica cada elemento entre renders** para saber cuál se mantuvo, cuál se movió y cuál se eliminó, sin volver a crear todo el DOM. Debe ser **único entre hermanos y estable** (que no cambie de un render a otro para el mismo dato). Aquí el nombre del modelo sirve porque es único. Lo que **no** debes hacer es usar el índice del `map` como key cuando la lista se reordena o filtra: si el orden cambia, React asocia el estado del elemento al índice equivocado y aparecen bugs sutiles (un input que conserva el texto de otra fila).

### 4.6 Estado derivado: se calcula en el render, NO en un efecto

*"Quiero mostrar 'eres usuario avanzado' cuando se enviaron 10+ mensajes. Esto NO es estado nuevo: es algo que se **deriva** del estado que ya tengo. El error de principiante es crear otro `useState` y un `useEffect` para 'sincronizarlo'. No. Lo calculo directo en el cuerpo del componente, en cada render."*

```tsx
function Panel({ titulo, modelos, porDefecto }: PanelProps) {
  const [enviados, setEnviados] = useState(0);

  // ✅ Estado derivado: una variable normal calculada en el render.
  // Siempre está sincronizada porque se recalcula cada vez que el componente se ejecuta.
  const esAvanzado = enviados >= 10;

  return (
    <section>
      <p>Mensajes: {enviados}</p>
      {esAvanzado && <span>Usuario avanzado</span>}
    </section>
  );
}
```

Graba esta frase: **si puedes calcularlo a partir de props y estado existentes durante el render, no es estado y no necesita un efecto.** Un `useEffect` que solo llama a `setOtroEstado` para mantenerlo "en sync" es casi siempre un error: agrega un render extra y una fuente de bugs.

### 4.7 Custom hook: extraer lógica reutilizable

*"El par `[enviados, setEnviados]` más la lógica de incrementar es algo que podría reutilizar en otro componente. Lo extraigo a un **custom hook**: una función cuyo nombre empieza por `use` y que puede llamar a otros hooks. No es magia; es solo una función que compone hooks."*

```tsx
// useContador.ts — un custom hook. La regla: el nombre DEBE empezar por "use".
import { useState, useCallback } from "react";

function useContador(inicial = 0) {
  const [valor, setValor] = useState(inicial);
  const incrementar = useCallback(() => setValor((n) => n + 1), []);
  const reiniciar = useCallback(() => setValor(inicial), [inicial]);
  return { valor, incrementar, reiniciar };
}

// En el componente, la lógica desaparece detrás de una línea legible:
function Panel(props: PanelProps) {
  const { valor: enviados, incrementar, reiniciar } = useContador(0);
  // ...usas enviados, incrementar, reiniciar
}
```

Un custom hook **no comparte estado** entre componentes: cada componente que lo llama obtiene su propia copia del estado. Lo que se comparte es la **lógica**, no los datos. (Para compartir datos entre componentes lejanos verás `useContext` en [4.8 Estado global](/fase-4-frontend/4-8-estado-global/).)

### 4.8 `useEffect`: solo para sincronizar con sistemas externos

*"Hasta aquí no usé ningún `useEffect`, y eso es buena señal. `useEffect` es para una sola cosa: **sincronizar tu componente con algo que vive fuera de React** —el título del documento, una suscripción a un WebSocket, un `fetch`—. Quiero que el título de la pestaña refleje los mensajes enviados. Eso sí es un efecto: estoy tocando el `document`, que es un sistema externo."*

```tsx
import { useEffect } from "react";

useEffect(() => {
  // 1) "setup": corre DESPUÉS de que React pinta la pantalla.
  document.title = `(${enviados}) Panel de IA`;
}, [enviados]); // 2) "deps": vuelve a correr solo si `enviados` cambió.
```

Tres cosas que debes interiorizar del array de dependencias:

- **Sin array** → el efecto corre **después de cada render**. Casi nunca es lo que quieres.
- **Array vacío `[]`** → corre **una sola vez al montar** (y limpia al desmontar).
- **`[enviados]`** → corre al montar y cada vez que `enviados` cambie (React compara con `Object.is`).

Y el caso que más se equivoca todo el mundo: un efecto que hace `fetch` necesita **cleanup** para no provocar una *race condition*. Pienso en voz alta: *"Si el usuario cambia de modelo rápido, se disparan dos fetch; el primero puede llegar después del segundo y pintar datos viejos. Devuelvo una función de limpieza que marca la respuesta vieja como 'ignórala'."*

```tsx
useEffect(() => {
  let ignorar = false;
  fetchInfoModelo(modelo).then((data) => {
    if (!ignorar) setInfo(data); // solo el fetch vigente actualiza el estado
  });
  return () => {
    ignorar = true; // cleanup: corre antes del próximo efecto y al desmontar
  };
}, [modelo]);
```

> Detalle de React 19 que confunde a todos: en desarrollo, con `<StrictMode>` activo, React **monta, desmonta y vuelve a montar** cada componente una vez. Verás tus efectos correr dos veces. No es un bug: es React presionando para que escribas el *cleanup* correctamente. En producción corre una sola vez.

## 5. Errores y malentendidos comunes

:::caution[Podrías pensar... pero está mal]

**"Si quiero que algo cambie cuando el estado cambia, va en un `useEffect`."**
Mal. Si lo puedes **calcular** durante el render, es estado derivado (sección 4.6) y va como variable normal. Si es una **respuesta a una interacción** (un clic envía analytics), va en el *event handler*, no en un efecto que observa el estado. `useEffect` es solo para sincronizar con sistemas **fuera** de React. Regla de Dan Abramov: *"si no estás sincronizando con un sistema externo, probablemente no necesitas un efecto."*

**"Muto el estado y luego llamo a `setX` para avisar."**
Mal y peligroso. `estado.push(x)` o `objeto.campo = y` mutan en sitio; React compara por **identidad de referencia** (`Object.is`) y, al ver el mismo objeto, puede no re-renderizar. Siempre crea una copia nueva: `setLista([...lista, nuevo])`, `setObj({ ...obj, campo: y })`.

**"`useMemo` y `useCallback` hacen mi app más rápida, los pongo en todo."**
Mal. Memoizar tiene un costo (guardar el valor, comparar deps). Solo valen la pena cuando: (a) el cálculo es **genuinamente caro** (filtrar 50.000 filas), o (b) pasas una **referencia** a un componente memoizado o al array de deps de un efecto y necesitas que sea estable. Para todo lo demás, el render normal de React ya es rápido. Y en 2026 el **React Compiler** (estable desde octubre 2025) memoiza automáticamente por ti, así que escribir `useMemo`/`useCallback` a mano es cada vez menos necesario. Mídelo antes de optimizar.

**"El índice del `map` sirve de `key`."**
Solo si la lista **nunca** se reordena, filtra ni inserta en medio. En cuanto eso pasa, el índice deja de identificar al mismo dato y React mezcla el estado de las filas (el caso clásico: filtras una lista de inputs y el texto salta de fila). Usa un id estable del dato.

**"Uso `useRef` para guardar estado que se ve en pantalla."**
Mal. Cambiar `ref.current` **no** dispara un re-render. `useRef` es para valores que persisten entre renders pero que **no** se muestran (un id de timer, el nodo del DOM de un input para hacerle `.focus()`). Si quieres que la UI reaccione, es `useState`.

:::

Un *non-example* que parece correcto pero no lo es —léelo y detecta el bug antes de seguir:

```tsx
// 🐛 ¿Qué tiene de malo?
function Buscador({ items }: { items: string[] }) {
  const [query, setQuery] = useState("");
  const [filtrados, setFiltrados] = useState(items);

  useEffect(() => {
    setFiltrados(items.filter((i) => i.includes(query)));
  }, [query, items]);

  return (
    <input value={query} onChange={(e) => setQuery(e.target.value)} />
    // ...renderiza `filtrados`
  );
}
```

El bug: `filtrados` es estado derivado disfrazado de estado real. Este código hace **dos renders** por cada tecla (uno por `setQuery`, otro por `setFiltrados`), puede pintar resultados desfasados un frame, y duplica la fuente de verdad. Lo correcto es borrar el `useState` de `filtrados` y el `useEffect`, y calcular `const filtrados = items.filter(i => i.includes(query))` directo en el render. Este es **exactamente** el músculo que entrenas en el ejercicio A.

### 6.1 Predice (antes de ejecutar nada)

Lee este componente y **predice en voz alta o en papel** qué imprime la pantalla tras hacer clic UNA vez en el botón:

```tsx
function Doble() {
  const [n, setN] = useState(0);
  function alClic() {
    setN(n + 1);
    setN(n + 1);
    console.log("n dentro del handler:", n);
  }
  return <button onClick={alClic}>n = {n}</button>;
}
```

Preguntas para tu predicción: (1) ¿qué número muestra el botón tras un clic? (2) ¿qué imprime el `console.log` en ese mismo clic? Escribe tus dos respuestas antes de seguir.

<Solucion title="Ver respuesta (después de predecir)">

(1) El botón muestra **`n = 1`**, no 2: las dos llamadas `setN(n + 1)` leen el mismo `n` (la foto del render actual, 0) y ambas calculan 1. (2) El `console.log` imprime **`0`**: dentro del handler, `n` sigue siendo la foto de **este** render; el nuevo valor solo existe en el **próximo** render. Para que suba a 2, usa `setN(prev => prev + 1)` en ambas (sección 4.4).

</Solucion>

### 6.2 Parsons: reordena para crear un input controlado

Estas líneas forman un input controlado con TypeScript, pero están **desordenadas**. Reescríbelas en el orden correcto (a mano, en papel):

```text
A)   return <input value={texto} onChange={(e) => setTexto(e.target.value)} />;
B) import { useState } from "react";
C)   const [texto, setTexto] = useState<string>("");
D) function CampoNombre() {
E) }
```

<Solucion title="Ver orden correcto (después de intentarlo)">

`B → D → C → A → E`:

```tsx
import { useState } from "react";
function CampoNombre() {
  const [texto, setTexto] = useState<string>("");
  return <input value={texto} onChange={(e) => setTexto(e.target.value)} />;
}
```

Lo esencial de un input **controlado**: el `value` viene del estado y el `onChange` lo actualiza. React es la única fuente de verdad del valor; por eso siempre van juntos. Si pones `value` sin `onChange`, el input queda de solo lectura.

</Solucion>

### 6.3 Modifica

Toma el `CampoNombre` de arriba y, sin ejecutarlo aún, modifícalo en papel para que: (a) muestre debajo `Hola, {texto}` solo si `texto` no está vacío (pista: estado derivado + render condicional con `&&`), y (b) tipe el evento explícitamente como `React.ChangeEvent<HTMLInputElement>`. ## 7. Ejercicios Primero-Sin-IA

Dos ejercicios complementarios. El A es de código (lo construyes y lo validas con tests); el B es de razonamiento (depuras y escribes tu diagnóstico). Las carpetas viven en tu repo: `ejercicios/fase-4/filtro-buscable-react/` y `ejercicios/fase-4/depurar-efectos-react/`. Ábrelas en tu editor.

<Reto title="A — Filtro buscable tipado (sin un solo useEffect)" timebox="40 min">

Construye un componente `FiltroBuscable` en React 19 + TypeScript que reciba una lista de strings por props y un input de búsqueda que la filtre en vivo, **sin distinguir mayúsculas**. La clave del ejercicio: el resultado filtrado es **estado derivado** —se calcula en el render— y por tanto **no debe haber ningún `useEffect` ni un segundo `useState` para la lista filtrada**.

Hecho significa:
- Props tipadas con una `interface` (`items: string[]`, `placeholder?: string`).
- Un input **controlado** (`value` + `onChange`) cuyo placeholder viene de la prop.
- La lista filtrada se calcula en el render (substring, case-insensitive contra la consulta).
- Se renderiza como `<ul>`/`<li>` con `key` estable (el propio string del item).
- Cuando ningún item coincide, se muestra un elemento con el texto exacto `Sin resultados`.
- Cero `useEffect`, cero estado redundante. Los tests (`pnpm install && pnpm test`) pasan en verde.
- Puedes explicar sin notas por qué este componente no necesita `useEffect`.

Sigue el ciclo Primero-Sin-IA: intenta a mano, luego consulta la [documentación oficial de React](https://react.dev/learn), y solo al final usa IA para *revisar*, no para *generar*.

<Solucion title="Pista (ábrela solo si te trabaste de verdad)">

El esqueleto es: un `useState<string>("")` para la consulta, una constante `const visibles = items.filter(i => i.toLowerCase().includes(query.toLowerCase()))` calculada en el cuerpo, y el JSX que mapea `visibles` a `<li key={i}>`. El render condicional de "Sin resultados" es `{visibles.length === 0 && <p>Sin resultados</p>}`. Nota que `visibles` se recalcula en cada render automáticamente: ahí está el punto del ejercicio. Esto es una pista, no la solución.

</Solucion>

<Reto title="B — Depura un panel lleno de useEffect" timebox="30 min">

Te entregamos `PanelBuggy.tsx`: un componente con **cinco** usos de `useEffect`. **Cuatro tienen un problema distinto** (estado derivado disfrazado de efecto, dependencia faltante, falta de cleanup que causa race condition, y un efecto que debería ser un event handler) y **uno está correcto**: tu trabajo es separarlos. **No lo ejecutes**: diagnostícalo leyendo, como en una revisión de código.

Hecho significa, en tu archivo `diagnostico.md`, para cada uno de los cinco efectos:
- Veredicto: ¿está bien o mal?
- Si está mal, **nombra el antipatrón exacto** (no "se ve raro").
- Explica la consecuencia observable (bucle infinito, render de más, datos viejos en pantalla, fuga de recursos, etc.).
- Describe el arreglo en palabras + el mínimo código necesario (no reescribas todo el componente).
- Para el que está bien, explica **por qué** es un uso legítimo de `useEffect`.

Reconocer el efecto correcto es tan importante como cazar los rotos: te entrena a no "ver bugs" donde no los hay. Es exactamente el ejercicio que harías en una entrevista técnica de React.

<Solucion title="Pista (ábrela solo si te trabaste de verdad)">

Para cada efecto pregúntate, en este orden: (1) ¿esto se podría **calcular en el render** en vez de guardarlo en estado? (estado derivado). (2) ¿el array de deps incluye **todo** lo que el efecto lee? (dependencia faltante). (3) ¿el efecto arranca algo asíncrono o una suscripción que habría que **cancelar**? (falta cleanup). (4) ¿esto ocurre como **respuesta a una interacción** del usuario y no como sincronización con un sistema externo? (debió ser event handler). Una de las preguntas da "está bien": no todos los efectos están rotos. Esto es una pista, no la solución.

</Solucion>
